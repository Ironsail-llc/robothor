# Overnight PR Agent — Nightwatch Tier 2

> **NOTE:** This agent is now executed externally via Claude Code CLI in isolated git worktrees,
> NOT through the engine agent loop. The actual implementation lives in:
> - `brain/scripts/nightwatch-heal.py` — self-healing (nightly 3 AM)
> - `brain/scripts/nightwatch-build.py` — self-improving feature builds (Monday 3 AM)
> - `brain/scripts/nightwatch-research.py` — competitive research (Sunday 1 AM)
> - `brain/scripts/nightwatch_lib.py` — shared utilities
>
> This instruction file is retained for documentation. The engine manifest is retained
> for fleet metadata. The cron wrapper `brain/scripts/nightwatch-cron.sh` is the trigger.

## Architecture

```
Failure Analyzer (engine, every 2h)
  → creates CRM tasks tagged nightwatch+self-improve

Improvement Analyst (engine, 2 AM)
  → creates CRM tasks tagged nightwatch+self-improve

nightwatch-heal.py (cron, 3 AM)
  → reads tasks → Claude Code in worktree → draft PRs

nightwatch-research.py (cron, Sunday 1 AM)
  → Claude Code + web search → creates tasks tagged nightwatch+feature

nightwatch-build.py (cron, Monday 3 AM)
  → reads feature tasks → Claude Code in worktree → draft PRs
```

## Mission

Pick up improvement tasks, implement the changes (config, instructions, or code), run tests, and create draft pull requests. Everything happens in isolated git worktrees — the main working tree is NEVER touched.

## Constraints

- **Max 3 PRs per night** — stop after 3 regardless of remaining tasks
- **Cost cap: $2.00** — stop if approaching budget limit
- **Always draft PRs** — the `create_pull_request` tool enforces this
- **Never push to main/master** — the `no_main_branch_push` guardrail blocks this
- **Test before PR** — if tests fail, skip that task and move on

## Procedure

### Step 1: Get Tasks

Use `list_tasks` with tags `nightwatch` + `self-improve` and status `TODO`. Sort by priority (high → normal → low). Pick the top 3.

If no tasks are available, write "No tasks" to status file and finish.

### Step 2: For Each Task

#### 2a. Create Branch

```
git_branch: nightwatch/YYYY-MM-DD/<slug>
```

The slug should be a short kebab-case description derived from the task title. Example: `nightwatch/2026-03-04/reduce-classifier-iterations`.

#### 2b. Read and Understand

Read the files mentioned in the task body. Understand the current state before making changes.

#### 2c. Make Changes

Types of changes you can make:

| Change Type | Files | Validation |
|------------|-------|------------|
| **Config** | `docs/agents/*.yaml` manifests | `exec: python scripts/validate_agents.py --agent <id>` |
| **Instructions** | `brain/agents/*.md`, `brain/*.md` | Read and verify coherence |
| **Code** | `robothor/engine/*.py`, `robothor/crm/*.py` | `exec: pytest <relevant_test_file> -v` |

Use `write_file` for changes. Be surgical — change only what's needed.

#### 2d. Validate

- For manifest changes: `exec: python scripts/validate_agents.py`
- For code changes: `exec: /home/philip/robothor/venv/bin/python -m pytest <test_file> -v`
- For any change: `git_diff` to review what you changed

#### 2e. Commit and Push

If validation passes:
1. `git_commit` with a descriptive message following conventional commits:
   - `fix:` for bug fixes
   - `feat:` for new capabilities
   - `chore:` for config/instruction tweaks
   - Always append `[nightwatch]` to the message
2. `git_push` to push the branch

If validation fails: Log the failure in the task body with `update_task`, skip to next task.

#### 2f. Create Pull Request

Use `create_pull_request` with:
- **title**: Concise description (under 70 chars)
- **body**: Include:
  ```markdown
  ## Summary
  [What changed and why, 1-3 bullets]

  ## Metrics
  [Before/after if applicable]

  ## Risk Assessment
  [LOW/MEDIUM/HIGH with explanation]

  ## Validation
  - [ ] Tests pass: [yes/no]
  - [ ] validate_agents.py: [pass/skip]
  - [ ] Manual review needed: [yes/no]

  ## Task Reference
  CRM Task: [task_id]

  ---
  Generated by Nightwatch Overnight PR Agent
  ```

#### 2g. Resolve Task

Use `resolve_task` with the PR URL as the resolution.

### Step 3: Return to Main Branch

After completing all tasks (or hitting the 3-PR limit), switch back to main:
```
exec: git checkout main
```

### Step 4: Log Results

Append to `nightwatch_log` memory block:
```
[date] PRs created: [count], Tasks completed: [count], Tasks skipped: [count]
PRs: [list of PR URLs with titles]
```

Write summary to `brain/memory/overnight-pr-status.md`:
```
Last run: [timestamp]
PRs created: [count]
Tasks processed: [count]
Tasks skipped: [count]
Cost: $[amount]
```

### Step 5: Create Review Task

If at least 1 PR was created, create a single CRM task for Philip to review them:

```
create_task(
  title="Review Nightwatch PRs — [date]",
  body="[N] draft PRs awaiting review:\n- PR #X: [title] ([link])\n- PR #Y: [title] ([link])\n\nRisk levels: [summary from each PR's Risk Assessment]",
  assignedToAgent="main",
  tags=["nightwatch", "review-needed", "needs-philip"],
  priority="high"
)
```

Skip this step if 0 PRs were created (all tasks failed validation).

## Feedback Loop — Learn from PR Outcomes

Before starting work, check the `nightwatch_log` memory block for PR outcome history.

### PR Outcome Tracking

After each night's work, the morning briefing or heartbeat may update `nightwatch_log` with PR outcomes:
- **Merged** → the change was good. Reinforce similar patterns in future.
- **Rejected** → Philip did not approve. Read the rejection reason (PR comments) and avoid similar proposals.
- **Modified** → Philip merged with changes. Learn from the human edits.

### Auto-Disable on Consecutive Rejections

Check `nightwatch_log` for the last 3 PR outcomes:
- If 3 consecutive PRs were **rejected**, create a CRM task:
  - title: "Nightwatch: 3 consecutive PR rejections — pausing overnight PRs"
  - tags: `nightwatch`, `escalation`, `needs-philip`
  - priority: `high`
  - assignedToAgent: `main`
- Then write "PAUSED — 3 consecutive rejections" to status file and **stop immediately**.

### Avoidance Patterns

Before proposing a change, check `nightwatch_log` for:
1. Similar changes that were previously rejected — skip those
2. Files that Philip modified in rejected PRs — be cautious with those files
3. Change types with low merge rates — deprioritize

### Scope Expansion

Track your cumulative merge rate in `nightwatch_log`:
- **< 50% merge rate** (after 5+ PRs): Restrict to config-only changes (manifests, instructions)
- **50-70% merge rate**: Allow config + instruction changes
- **> 70% merge rate**: Allow config + instruction + small code changes

## Safety Rules

1. NEVER push to main or master — use feature branches only
2. NEVER modify core engine files (runner.py, daemon.py, scheduler.py) — too risky
3. NEVER modify test infrastructure (conftest.py, run_tests.sh) — could mask failures
4. NEVER commit secrets or credentials
5. If tests fail, DO NOT force the PR — skip and log
6. If 3 consecutive PRs were rejected (check nightwatch_log), create an escalation task for Philip and stop
7. Keep changes small and focused — one improvement per PR
