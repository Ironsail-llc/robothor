## Memory Intelligence Pipeline Overhaul

### Problem Discovered
- Memory Intelligence cron (Moltbot agentTurn + Qwen3-Next) was failing
- Qwen3-Next can't do agent tool-calling — puts everything in "thinking" field
- All Qwen3 models (80B, 14B, etc.) have this same contamination issue
- Model "thinks when it should act" — generates 200+ tokens deliberating, never executes

### Solution Implemented
- **Architecture:** Python script + system cron (not Moltbot agentTurn)
- Python handles all orchestration (file I/O, API calls, DB writes)
- LLM called only for generation tasks (no tool-calling needed)
- Schedule: 3:30 AM daily via system crontab

**Issue:** Philip missed 3 morning meetings (9:00, 9:45, 10:00 AM) because I wasn't alerting him.

## Canonical Cron Architecture Finalized (12:26 PM)

Created CRON_DESIGN.md as the permanent, canonical specification.

**The Rule:**
- Every cron reads logs → does its job → updates its log
- "Escalate" = write `escalated: true` to log (NOT message main channel)
- Heartbeat reads all logs → synthesizes → surfaces escalations → finalizes

**Log Entry Lifecycle:**
1. NEW: processedAt set by cron
2. PROCESSED: actionTaken, escalated flag set
3. SURFACED: surfacedAt set by heartbeat when reported to Philip
4. FINALIZED: finalizedAt set when verified complete

**Updated crons to follow this design:**
- Calendar Check
- Email Processing  
- Jira Sync
- Heartbeat

**This is FINAL. Do not deviate without Philip's approval.**
